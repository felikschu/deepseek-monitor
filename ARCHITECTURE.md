# DeepSeek 监控系统架构设计

## 1. 系统概述

### 1.1 设计目标

设计一个**轻量级、自动化、可扩展**的监控系统，能够：
- 检测前端资源变化（JS/CSS hash）
- 提取和比较模型配置
- 监控行为特征变化
- 生成可读的报告

### 1.2 设计原则

1. **非侵入式**：只通过公开的网页端访问，不使用私有 API
2. **低频次**：避免频繁请求，防止被封禁
3. **模块化**：各监控模块独立，易于扩展
4. **可维护**：清晰的代码结构和文档

### 2.2 模块职责

#### 2.2.1 控制层

**DeepSeekMonitor (主控制器)**

职责：
- 初始化和协调所有模块
- 管理监控生命周期
- 处理用户命令

主要方法：
```python
async def initialize()          # 初始化所有监控器
async def run_full_check()      # 执行完整检查
async def run_continuous_monitoring()  # 持续监控
```

#### 2.2.2 监控模块层

**FrontendMonitor (前端监控器)**

职责：
- 检测 JS/CSS 文件 hash 变化
- 提取关键代码模式
- 识别新功能

监控流程：
```
1. 获取主页 HTML
2. 解析出前端资源 URL
3. 下载资源并计算 hash
4. 与历史 hash 比较
5. 提取代码模式
6. 检测新功能
```

**ConfigMonitor (配置监控器)**

职责：
- 提取运行时配置
- 监控 Feature Flag
- 检测 API 端点变化

监控流程：
```
1. 启动无头浏览器
2. 访问 DeepSeek 网页
3. 执行 JS 提取配置
4. 监听网络请求
5. 分析配置差异
```

**BehaviorMonitor (行为监控器)**

职责：
- 执行标准测试用例
- 分析响应特征
- 检测行为异常

监控流程：
```
1. 启动浏览器
2. 逐个执行测试用例
3. 记录响应时间和内容
4. 与历史基线比较
5. 检测异常
```

#### 2.2.3 告警模块

**Alerter (告警器)**

职责：
- 收集变化信息
- 格式化告警消息
- 发送通知

告警类型：
- 控制台告警（默认）
- 邮件告警（待实现）
- Webhook 告警（待实现）

#### 2.2.4 报告模块

**Reporter (报告生成器)**

职责：
- 生成监控报告
- 统计分析
- 提供建议

报告格式：
- Markdown（可读）
- JSON（机器可读）

#### 2.2.5 数据访问层

**StorageManager (存储管理器)**

职责：
- 管理数据库连接
- 提供数据访问接口
- 清理历史数据

数据表：
```sql
resource_hashes    -- 资源 hash 历史
code_patterns      -- 代码模式历史
model_configs      -- 模型配置历史
api_endpoints      -- API 端点历史
test_results       -- 测试结果历史
check_results      -- 检查结果历史
detected_features  -- 已检测功能
changes_history    -- 变化历史
```

## 3. 数据流设计

### 3.1 监控检查流程

```
开始
  │
  ▼
初始化监控器
  │
  ├─► FrontendMonitor.check()
  │   ├─ 获取主页 HTML
  │   ├─ 解析资源 URL
  │   ├─ 下载并计算 hash
  │   ├─ 提取代码模式
  │   └─ 返回变化
  │
  ├─► ConfigMonitor.check()
  │   ├─ 启动浏览器
  │   ├─ 提取配置
  │   ├─ 监听 API
  │   └─ 返回变化
  │
  ├─► BehaviorMonitor.check()
  │   ├─ 启动浏览器
  │   ├─ 执行测试用例
  │   ├─ 分析响应
  │   └─ 返回变化
  │
  ▼
合并所有变化
  │
  ├─► StorageManager.save_check_results()
  │
  ├─► Alerter.process_and_alert()
  │
  ▼
结束
```

### 3.2 数据存储策略

**实时数据**（SQLite）
- 资源 hash
- 代码模式
- 配置快照
- 测试结果

**归档数据**（JSON）
- 完整报告
- 大型快照

## 4. 变化检测策略

### 4.1 前端资源变化

**检测方法**：
1. 计算 JS/CSS 文件的 MD5 hash
2. 与上次保存的 hash 比较
3. 如果不同，标记为变化

**变化类型**：
- `resource_change`: 文件 hash 变化
- `new_resource`: 新增资源
- `pattern_change`: 代码模式变化

### 4.2 配置变化

**检测方法**：
1. 深度比较配置对象
2. 记录所有差异路径

**变化类型**：
- `config_change`: 配置值变化
- `api_endpoints_change`: API 端点增减

### 4.3 行为变化

**检测方法**：
1. 计算历史基线（平均值、中位数）
2. 比较当前指标与基线
3. 标记显著偏差

**异常阈值**：
- 响应时间 > 基线 × 2
- 响应长度变化 > 50%

## 5. 性能考虑

### 5.1 请求频率

**建议**：
- 前端检查：每 3-4 小时
- 配置检查：每 3-4 小时
- 行为检查：每 6-12 小时（较慢）

**原因**：
- 避免触发反爬机制
- 减少服务器负担
- 节省资源

### 5.2 并发控制

```python
# 限制并发请求数
max_concurrent_checks: 2
```

### 5.3 数据清理

```python
# 定期清理旧数据
retention_days: 90
```
